##### OWASP Top 10:2021: A01:2021 - Broken Access Control

- Noussut OWASP-listan ykköseksi: 94 % testatuista sovelluksista löytyi pääsynhallintapuute
- Käyttäjä pääsee tekemään yli oikeuksiensa (katsoo, muokkaa tai poistaa toisten dataa)
- Tyypilliset reiät: IDOR (URLin id-vaihto), force browsing, JWT-peukalointi, puuttuvat tarkistukset POST/PUT/DELETE:ssä, privilege escalation, CORS-virhekonffit
- Suojaus: deny by default, tarkistukset vain palvelinpuolella, record ownership -mallit, lokitus + rate limit, lyhytikäiset JWT:t
- Olen pyrkinyt ottamaan tämän huomioon yhden kurssin lopputyössäni varmistamalla, että pyynnön userId täsmää JWT:n käyttäjään – pelkkä roolitarkistus ei riitä estämään IDOR-tyyppistä hyökkäystä

##### PortSwigger Academy: Insecure direct object references (IDOR)
- IDOR = pääsynhallinnan haavoittuvuus, jossa sovellus käyttää käyttäjän syöttämää arvoa suoraan objektin hakemiseen ilman omistajuustarkistusta
- Termi vakiintui OWASP Top 10:2007 -listalta; liittyy useimmin horisontaaliseen oikeuksien laajentamiseen (toisen samantasoisen käyttäjän data), mutta myös vertikaaliseen (admin-oikeudet)
- Esimerkit:
    - Tietokantaviittaus: ?customer_number=132355 → numeroa vaihtamalla näkee toisen asiakkaan tiedot
    - Staattinen tiedosto: /static/12144.txt → tiedostonimen vaihtamalla pääsee käsiksi toisten chat-lokeihin tms.


 ##### PortSwigger Academy: Path traversal
- Hyökkääjä lukee (joskus kirjoittaa) palvelimen tiedostoja syöttämällä polkuun ../-sekvenssejä – tunnetaan myös nimellä directory traversal
- sovelluskoodi, tunnukset, käyttöjärjestelmätiedostot paljastuvat -> pahimmillaan koko palvelin saadaan haltuun
- Suojauskeinot: Absoluuttinen polku, Sisäkkäiset sekvenssit, URL-enkoodaus tai tupla-enkoodaus

##### PortSwigger Academy: Cross-site scripting
- Hyökkääjä saa haavoittuvan sivuston palauttamaan haitallista scriptiä, joka suoritetaan uhrin selaimessa → kiertää same-origin policyn ja voi esiintyä uhrin nimissä
- Kolme tyyppiä: Reflected XSS -> haitallinen skripti tulee suoraan HTTP-pyynnöstä ja heijastuu vastaukseen, Stored XSS –> skripti tallentuu palvelimen tietokantaan (esim. kommenttiin) ja suoritetaan kaikilla sivun katsojilla, DOM-based XSS – haavoittuvuus on selainpuolen JavaScriptissä, joka kirjoittaa epäluotettavaa dataa DOM:iin
- Hyökkääjä voi esiintyä uhrina, varastaa tunnuksia tai injektoida troijalaisen
- Suojaus: Suodata syöte saapuessaan, Encode output, Content Security Policy (CSP)

Loin uuden sertifikaatin ja tallensin sen tiedostona koneelleni. Tämän jälkeen avasin firefoxin ja menin **asetuksiin ,sieltä kirjoitin hakuun "certificate -> View Certiicates, siellä import ja otin filen minkä tallensin tiedostoihin, klikkasin "Trust this CA to identify websites"**, tarkastin ja se on listassa:

<img width="1324" height="924" alt="image" src="https://github.com/user-attachments/assets/86906d24-d2de-40f8-82f8-66d890ccc9ba" />

Sitten menin takaisin asetuksien alkusivuun ja kirjoitin hakuun **proxy sitten Network settings -> settings, siellä valitsin Manual proxy configuration, HTTP Proxy 127.0.0.1, Port 8080, ja rastin "Also use this proxy for HTTPS". Asetin myös `about:config` -sivulla `network.proxy.allow_hijacking_localhost = true`, sillä Firefox ei muuten ohjaa localhost-liikennettä proxyn läpi.**

Testasin niin http toimi, mutta ensimmäisellä HTTPS-testillä yle.fi antoi "Did Not Connect: Potential Security Issue" -virheen HSTS-käytännön vuoksi. Syy oli siinä, että Firefox ei luottanut ZAPin sertifikaattiin, vaikka se näkyi Authorities-listassa. Tutkin onglemaa ja ratkaisin sen poistamalla vanhan sertifikaatin, vein sen ZAPista uudelleen ja tuotin Firefoxiin varmistaen että "Trust this CA to identify websites" -rasti oli oikeasti päällä. Tämän jälkeen HTTPS-sivut latautuivat normaalisti ilman varoituksia.

Laitoin vielä kuvien sieppauksen päälle: **ZAP:issa Tools → Options → Display → Process images in HTTP requests/responses**, tämä ei oletuksena näytä kuvapyyntöjä Historyssä.
